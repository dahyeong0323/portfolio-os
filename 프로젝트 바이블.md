# Portfolio OS Bible v1.0  
## Ledger-First Personal Investment Operating System

> 이 문서는 Portfolio OS 프로젝트의 기준 문서다.  
> 앞으로 설계, 구현, 프롬프트, 코드, 기능 추가, 의사결정은 이 문서를 기준으로 판단한다.  
> 목적은 “AI가 대신 투자하는 봇”을 만드는 것이 아니라, 공격적인 장기 투자자가 감정·편향·장부 오류·수치 환각·과매매·API 리스크를 통제하면서 더 나은 투자 의사결정을 하도록 돕는 **개인 투자 운영체제**를 만드는 것이다.

---

## 0. 최종 결론

이 프로젝트는 만들 가치가 있다.

단, 만들 대상은 다음이 아니다.

- 자동매매 봇
- 종목 추천 챗봇
- 매주 매수 신호를 주는 알림 봇
- AI가 인간 대신 판단하고 실행하는 시스템
- 모든 데이터를 LLM에게 넣고 답을 받는 무거운 리포트 생성기

만들 대상은 다음이다.

> **Ledger-first, Risk-engine-gated, Macro-aware, Human-approved Personal Portfolio OS**

즉:

1. 장부가 먼저다.
2. 장부가 맞지 않으면 공식 추천은 없다.
3. 리서치는 반대 논리를 강제한다.
4. Senior Agent는 판단하지만 계산하지 않는다.
5. Risk Engine이 수치적 한도를 코드로 검증한다.
6. Trading Agent는 주문 티켓을 만들 뿐, MVP에서는 주문을 실행하지 않는다.
7. 인간이 최종 승인하고, 초기에는 직접 MTS/HTS에서 주문한다.
8. 우회 거래는 금지하는 대신 Declared Override로 시스템 안에 기록한다.
9. 전체 포트폴리오의 macro/correlation risk를 별도 레이어로 본다.
10. LLM 프롬프트와 모델 품질은 QA 대상이다.
11. 토큰 비용과 장기 기억은 Context Economy/RAG 구조로 관리한다.

## 1. 프로젝트의 존재 이유

### 1.1 문제 정의

공격적인 개인 투자자는 보수적인 투자자와 다른 문제를 가진다.

보수적인 투자자의 핵심 문제는 보통 “손실을 줄이고 안정성을 높이는 것”이다.  
하지만 공격적인 장기 투자자의 핵심 문제는 다르다.

공격적 투자자는 큰 upside를 추구한다.  
BTC, 레버리지 ETF, 고성장주, venture-style public equity basket 같은 자산을 활용할 수 있다.  
문제는 공격성 자체가 아니다.

진짜 문제는 다음이다.

- 강한 확신이 확증편향으로 변한다.
- 손실 경험이 revenge trading으로 이어진다.
- 큰 수익 후 세금 준비를 못 해 강제 매도가 발생한다.
- 레버리지 상품이 전체 순자산을 지배한다.
- 현금/부채/세금/투자 포지션이 분리되어 전체 리스크가 안 보인다.
- 시장 폭락장에서 모든 자산의 상관관계가 1에 가까워진다.
- AI가 그럴듯하게 말하지만 숫자 계산을 틀릴 수 있다.
- 시스템이 너무 빡빡하면 인간이 시스템 밖에서 거래한다.
- 장부와 실제 계좌가 틀어지면 모든 판단이 쓰레기 데이터 기반이 된다.

Portfolio OS는 이 문제를 해결하기 위한 시스템이다.

### 1.2 핵심 철학

> 공격성을 없애는 것이 목표가 아니다.  
> 공격성을 구조화하는 것이 목표다.

이 프로젝트는 사용자를 보수적인 투자자로 바꾸기 위한 시스템이 아니다.  
BTC 확신, 고성장주 upside, 장기 복리, 젊은 나이의 시간 우위는 유지한다.

다만 다음 네 가지가 전체 전략을 파괴하지 못하게 만든다.

1. 하나의 테마
2. 하나의 레버리지 상품
3. 하나의 세금 이벤트
4. 하나의 유동성 문제

## 2. 시스템의 최종 아키텍처

최종 구조는 단순 3-Agent 구조가 아니다.  
토론 후 확정된 구조는 다음과 같다.

```text
1. Portfolio Ledger
2. Reconciliation Engine
3. Research Agent
4. Anti-thesis QA
5. Macro & Correlation Risk Layer
6. Senior Investor Agent
7. Risk / Constraint Engine
8. Trading Agent
9. Escape Hatch / Override System
10. Decision Journal / Post-Mortem
11. Model QA / Prompt Governance Layer
12. Context Economy / RAG / Token Budget Layer
13. Human Final Decision
```

요약하면:

```text
Ledger
 -> Reconciliation
 -> Research + Anti-thesis QA
 -> Macro / Correlation Layer
 -> Senior Investor Agent
 -> Risk / Constraint Engine
 -> Trading Agent
 -> Human Approval
 -> Manual Execution
 -> Temporary Log
 -> Reconciliation
 -> Decision Journal
```

이 구조의 핵심은 각 구성요소가 서로를 견제한다는 점이다.

```text
Research Agent는 Senior Agent의 판단 재료를 제공한다.
Anti-thesis QA는 Research Agent의 확증편향을 견제한다.
Macro Layer는 개별 자산 리서치가 놓치는 포트폴리오 전체 리스크를 견제한다.
Risk Engine은 Senior Agent의 수치적 환각을 견제한다.
Trading Agent는 검증된 주문 티켓만 전달한다.
Human은 최종 승인권을 가진다.
Override System은 인간의 우회 본능을 시스템 안에 흡수한다.
Reconciliation Engine은 장부와 실제 계좌의 불일치를 막는다.
Model QA는 LLM의 조용한 열화를 감시한다.
Context Economy는 토큰 비용과 기억 폭발을 막는다.
```

## 3. 절대 원칙

### 3.1 Ledger-first Principle

Portfolio OS의 심장은 AI가 아니라 장부다.

Research Agent나 Senior Agent보다 먼저 필요한 것은 정확한 원장이다.

공식 장부에는 다음이 포함된다.

- 보유 종목
- 수량
- 평균단가
- 현재가
- 평가금액
- 현금
- 부채
- 세금 준비금
- 수동 주문 기록
- 미확정 체결
- 계좌 대조 상태
- 마지막 reconciliation 날짜

### 3.2 No Sync, No Official Recommendation

장부가 실제 계좌와 맞지 않으면 공식 추천은 없다.

```text
ledger_status != reconciled
=> no official recommendation
```

단, 이것은 “아무 행동도 불가능”이라는 뜻이 아니다.

정확한 규칙은 다음이다.

```text
No sync -> no official recommendation
No sync -> emergency override ticket may still be created
```

즉, 시스템은 공식 추천의 권위는 지키되, 인간의 예외 행동은 시스템 안으로 흡수한다.

### 3.3 Research Agent는 판단하지 않는다

Research Agent는 사실만 수집한다.

금지 표현:

- buy
- sell
- attractive
- should add
- good opportunity
- undervalued
- overvalued
- strongly recommended

허용 표현:

- 가격이 얼마 변했다
- 매출이 몇 % 증가했다
- cash runway가 얼마다
- dilution risk가 있다
- ETF flow가 순유입/순유출이다
- thesis를 지지하는 사실
- thesis를 약화시키는 사실
- 확인하지 못한 데이터

### 3.4 Anti-thesis Mandatory

모든 자산 리서치에는 반대 논리가 반드시 있어야 한다.

각 자산별 Research Packet에는 최소한 다음이 포함되어야 한다.

```text
Bull facts
Bear facts
Neutral facts
Thesis-supporting facts
Thesis-challenging facts
Missing data
Source list
```

Anti-thesis가 없으면 보고서는 invalid 처리한다.

```text
Research report status: INVALID
Reason: Anti-thesis section missing
```

### 3.5 LLM judges; code calculates

LLM은 투자 논지, 시나리오, 우선순위 판단에 사용한다.  
하지만 숫자 계산, 현금 한도, 최대 비중, 세금 버퍼, 주간 매수 한도는 LLM에게 맡기지 않는다.

```text
Senior Agent = judgment
Risk Engine = calculation and hard constraints
```

### 3.6 Broker API는 MVP에서 제외한다

초기 버전에서는 broker API 자동 실행을 하지 않는다.

MVP의 실행 방식:

```text
Trading Agent creates validated order ticket
Human approves
Human manually executes on MTS/HTS
Human logs execution
System marks it provisional
Next reconciliation confirms it
```

### 3.7 인간 최종 승인

어떤 trade도 인간 승인 없이 실행되어서는 안 된다.

MVP에서는 인간이 직접 MTS/HTS에서 주문한다.

나중에 API 실행을 붙이더라도:

- explicit approval
- final order confirmation
- daily/weekly trade limit
- no market order by default
- no repeated automatic order
- audit trail

이 조건을 만족해야 한다.

### 3.8 Override는 금지하지 않고 기록한다

인간은 시스템을 우회할 수 있다.  
우회를 완전히 막는 것은 비현실적이다.

따라서 Shadow Trading을 방지하기 위해 “Declared Override” 기능을 둔다.

```text
Human wants to trade outside official recommendation
=> create Override Ticket
=> show risk warning
=> mark as human override
=> require later reconciliation
=> require post-mortem
```

# 4.1 Portfolio Ledger

## 역할

Portfolio Ledger는 시스템의 공식 원장이다.  
모든 리서치, 판단, 리스크 계산은 이 장부를 기준으로 한다.

## 저장 대상

초기 MVP에서는 CSV 또는 SQLite로 충분하다.

권장 테이블:

```text
holdings
transactions
cash_balances
liabilities
tax_reserves
manual_execution_logs
reconciliation_snapshots
decision_journal
override_logs
risk_rules
asset_theses
```

## Ledger 상태

모든 장부 데이터에는 상태가 있어야 한다.

| 상태 | 의미 | 공식 추천 가능 여부 |
|---|---|---|
| reconciled | 실제 계좌와 대조 완료 | 가능 |
| provisional | 수동 입력됐지만 아직 계좌 대조 전 | 제한적 가능 |
| stale | 일정 기간 이상 대조 안 됨 | 공식 추천 불가 |
| broken | 불일치 발견 | 공식 추천 불가 |

예시:

```text
Cash: 1,200,000 KRW
Data status: provisional
Last reconciled: 2026-05-10
Unconfirmed manual trades: 2
```

# 4.2 Reconciliation Engine

## 역할

실제 계좌 상태와 Portfolio OS 장부를 대조한다.

## 입력

- OS holdings
- OS transactions
- OS cash balance
- broker/account snapshot
- manual execution logs
- FX rate basis
- fees
- liabilities
- tax reserves

## 출력

```text
reconciliation_status: passed / failed / needs_review
ledger_status: reconciled / provisional / stale / broken
differences:
  - ticker
  - expected_quantity
  - actual_quantity
  - expected_cash
  - actual_cash
  - difference
```

## 규칙

```text
If reconciliation failed:
    block official recommendation
    allow only review mode or override mode
```

## MVP 구현 방식

초기에는 broker API를 쓰지 않는다.

가능한 입력 방식:

1. 사용자가 증권사 보유 내역 CSV 업로드
2. 사용자가 계좌 스냅샷을 수동 입력
3. 사용자가 MTS 캡처를 업로드하고 나중에 수동 정리
4. Google Sheet에 실제 계좌 스냅샷 입력

처음에는 자동 OCR보다 수동 CSV/Sheet 입력이 더 안전하다.

# 4.3 Research Agent

## 역할

Research Agent는 애널리스트다.  
판단하지 않고 사실만 수집한다.

## 입력

- reconciled portfolio state
- watchlist
- thesis files
- market data
- company data
- macro data
- news
- filings
- previous research state

## 출력

각 자산별 structured research packet:

```text
Asset: RKLB

1. Price and performance
2. Recent news
3. Financial metrics
4. Bull facts
5. Bear facts
6. Neutral facts
7. Thesis-supporting facts
8. Thesis-challenging facts
9. Missing data
10. Source list
11. Research status: valid / invalid
```

## 중요한 제한

Research Agent는 아래를 하지 않는다.

- 매수 추천
- 매도 추천
- 비중 추천
- valuation judgment
- “싸다/비싸다” 결론
- 감정적 표현

---

# 4.4 Anti-thesis QA

## 역할

Research Agent가 확증편향 수집기가 되는 것을 막는다.

## QA 기준

| 항목 | 통과 기준 |
|---|---|
| Bear facts | 각 자산별 최소 2개 이상 |
| Missing data | 확인 못 한 항목 명시 |
| Source diversity | 가격, 뉴스, 공시, 재무, macro 구분 |
| Judgment language | buy/sell/attractive 등 금지 |
| Thesis challenge | 기존 thesis를 약화시키는 사실 포함 |

## 실패 시

```text
Research QA: FAILED
Action: Do not pass to Senior Agent
```

# 4.5 Macro & Correlation Risk Layer

## 역할

개별 자산 리스크가 아니라 포트폴리오 전체 리스크를 본다.

BTC, QLD, 고성장주 바스켓은 서로 다른 자산처럼 보이지만 실제로는 다음 factor에 동시에 노출될 수 있다.

- global liquidity
- risk-on sentiment
- US growth equity
- real rates
- USD liquidity
- leverage
- crypto cycle
- duration risk

## 주요 출력

```text
Portfolio factor diagnosis:
- Risk-on exposure: extreme / high / medium / low
- Liquidity sensitivity: high
- BTC-related exposure: high
- Nasdaq/growth exposure: high
- Correlation stress: elevated
- Defensive buffer: low
- Macro regime: normal / stress / crisis
```

## 필수 지표

- BTC vs QQQ rolling correlation
- QLD vs high-growth basket correlation
- portfolio vs BTC beta
- portfolio vs Nasdaq beta
- average risk-asset correlation
- drawdown from recent high
- volatility spike
- risk-on/risk-off regime
- cash buffer ratio

## Macro Regime

| Regime | 의미 | 행동 |
|---|---|---|
| Normal | 시장 정상 | 일반 weekly review |
| Stress | 변동성/상관관계 상승 | 신규 매수 축소, cash reserve 상향 |
| Crisis | 유동성 충격 | Crash Playbook만 허용 |

---

# 4.6 Crash Playbook

## 역할

폭락장에서 즉흥적인 영끌 매수를 막고, 사전에 정한 방식으로 공격성을 구조화한다.

## 원칙

```text
폭락장은 기회일 수 있다.
하지만 기회는 즉흥이 아니라 사전 playbook으로만 잡는다.
```

## 예시 구조

```text
Trigger A:
- Nasdaq -10% from recent high
- BTC -20% from recent high
- portfolio drawdown > -15%
Action:
- deploy max 10% of available crash cash

Trigger B:
- Nasdaq -20%
- BTC -35%
- portfolio drawdown > -25%
Action:
- deploy additional 15% of available crash cash

Trigger C:
- extreme panic / liquidity shock
Action:
- deploy only if hard cash floor remains protected
```

## 절대 금지

- tax reserve 사용
- 부채 기반 emergency buy
- 현금 바닥 붕괴
- risk rule bypass without override log

# 4.7 Senior Investor Agent

## 역할

Senior Investor Agent는 CIO/PM 역할을 한다.  
Research Agent의 fact packet과 Macro Layer의 포트폴리오 진단을 바탕으로 판단한다.

## 할 수 있는 것

- thesis status 판단
- 시나리오별 해석
- 우선순위 결정
- no-action alternative 제시
- watchlist 우선순위 제시
- proposed action draft 생성

## 할 수 없는 것

- 최종 주문 생성
- Risk Engine 없이 매수 금액 확정
- 세금/현금 한도 직접 계산
- broker execution
- Research Agent 없이 판단
- invalid research 기반 신규 매수 추천

## Senior Memo 필수 섹션

```text
1. Executive summary
2. Portfolio-level diagnosis
3. Macro/correlation risk interpretation
4. Cash and liquidity interpretation
5. Asset-level thesis status
6. Proposed action draft
7. No-action alternative
8. Strongest opposing argument
9. What would change the decision
10. Required Risk Engine validation
```

## 출력 예시

```text
Preferred action: small add to QLD
Confidence: medium
No-action alternative: acceptable
Main risk: correlation spike and liquidity shock
Requires Risk Engine validation: yes
```

# 4.8 Risk / Constraint Engine

## 역할

Senior Agent의 제안을 deterministic code로 검증한다.

## 핵심 원칙

```text
If uncertain, reject.
If data missing, reject.
If ledger stale, reject official recommendation.
If tax reserve unknown, reject.
If cash uncertain, reject.
```

## 검증 항목

- minimum cash reserve
- weekly buy limit
- daily buy limit
- max asset weight
- max bucket weight
- BTC-related exposure
- QLD/leveraged ETF exposure
- high-growth basket max position size
- tax reserve protection
- debt exposure
- legacy MSTU rule
- override max size
- crash playbook limits

## 출력

```text
Risk Validation Result: PASSED / REJECTED / ADJUSTED

Proposed order:
- Buy QLD 300,000 KRW

Failed checks:
1. Minimum cash reserve violation
2. Weekly buy limit violation

Maximum allowed order:
- QLD up to 120,000 KRW
```

## 중요한 설계

Trading Agent는 Senior Agent의 텍스트를 직접 신뢰하지 않는다.  
Trading Agent는 Risk Engine이 검증한 structured JSON만 신뢰한다.

# 4.9 Trading Agent

## 역할

Trading Agent는 front-office communicator다.  
MVP에서는 broker API 실행자가 아니다.

## 기능

- validated order ticket 생성
- Telegram으로 요약 전달
- 인간 승인 요청
- manual execution 입력 받기
- provisional log 생성
- reconciliation 필요 표시
- confirmation message 발송

## Telegram 메시지 예시

```text
[Weekly Portfolio Review]
```

# 4.10 Escape Hatch / Override System

## 역할

시스템이 너무 빡빡해서 인간이 몰래 거래하는 Shadow Trading을 막는다.

## 원칙

> 우회를 없애려 하지 말고, 우회를 시스템 안으로 끌어들여 기록 가능한 예외로 만든다.

## Override 종류

| 유형 | 설명 |
|---|---|
| Emergency Buy Override | 폭락장 기회 매수 |
| Manual Correction Override | 장부/입력 오류 수정 |
| No-sync Override | 장부 미대조 상태에서 인간이 거래 원함 |
| Thesis Conviction Override | 시스템은 보류지만 인간 확신으로 거래 |
| Panic Override | 감정적 거래 위험 표시 필요 |

## Override Ticket 필수 항목

```text
1. Override reason
2. Asset
3. Side
4. Requested amount
5. System status
6. Risk warning
7. Max recommended emergency size
8. Human final choice
9. Mandatory reconciliation deadline
10. Mandatory post-mortem date
```

## 원칙

Override는 허용된다.  
하지만 기록 없이 우회하는 것은 실패다.

# 4.11 Decision Journal / Post-Mortem

## 역할

Portfolio OS를 단순 기록기가 아니라 학습 시스템으로 만든다.

## 모든 결정 기록

```text
decision_id
date
portfolio state
research packet id
senior memo id
risk validation id
human decision
approved / rejected / modified / override
execution status
reason
emotional state if relevant
```

## Post-Mortem 항목

```text
1. 당시 결정은 무엇이었는가?
2. 당시 근거는 무엇이었는가?
3. 반대 논리는 무엇이었는가?
4. 결과는 어땠는가?
5. thesis가 맞았는가?
6. 운이었는가 실력이었는가?
7. risk rule은 지켜졌는가?
8. 다음 결정에 반영할 교훈은 무엇인가?
```

## Review Cadence

| 주기 | 목적 |
|---|---|
| Weekly | 장부 확인, 리스크 점검, 소액 행동 여부 |
| Monthly | 배분 점검, 성과 복기, 현금 계획 |
| Quarterly | thesis 재평가, 세금/유동성/position size 재검토 |

# 4.12 Model QA / Prompt Governance Layer

## 역할

LLM 모델과 프롬프트의 조용한 열화(prompt drift, degradation)를 감시한다.

## 핵심 원칙

```text
Models are not trusted.
Prompts are not trusted.
Outputs are not trusted until QA passes.
```

## 관리 대상

- Research Agent prompt
- Senior Agent prompt
- Trading Agent prompt
- output schema
- model version
- eval results
- canary test results

## Golden Test Set

고정 테스트 케이스를 만든다.

예:

```text
Case 1: BTC bullish news + ETF outflow
Expected: bull/bear facts both present

Case 2: QLD drawdown + insufficient cash
Expected: no aggressive buy, requires Risk Engine

Case 3: RKLB contract news + dilution risk
Expected: both positive event and anti-thesis shown
```

## Canary Run

실제 weekly review 실행 전 작게 테스트한다.

```text
Step 0: Canary QA
- Research Agent benchmark
- Senior Agent benchmark
- schema validation
- anti-thesis validation
- hallucinated number check
- safety-phrase overuse check
```

실패 시:

```text
Weekly review blocked.
Reason: Model QA failed.
Action: use previous stable prompt/model or manual review.
```

## Prompt Versioning

```text
agents/
  research_agent.v1.0.md
  senior_agent.v1.0.md
  trading_agent.v1.0.md

evals/
  research_golden_set.jsonl
  senior_golden_set.jsonl

qa_results/
  2026-05-16-model-qa.json
```

각 리포트에는 다음 메타데이터를 붙인다.

```text
Model:
Prompt version:
QA score:
Canary status:
Generated at:
```

# 4.13 Context Economy / RAG / Token Budget Layer

## 역할

토큰 비용 폭발과 장기 기억 문제를 해결한다.

## 원칙

> LLM에게 모든 것을 읽히지 않는다.  
> 필요한 정보만 선별해서 넣는다.

## 정보 계층

```text
Layer 1: Raw Data Archive
- filings
- news
- price history
- account snapshots
- screenshots
- CSVs

Layer 2: Structured Extracts
- revenue growth
- margins
- cash runway
- ETF flows
- correlation metrics
- drawdown
- valuation metrics

Layer 3: Thesis Memory
- original thesis
- current thesis status
- invalidation triggers
- key updates
- last major changes

Layer 4: Decision Memory
- past recommendations
- approved/rejected trades
- override events
- post-mortems
- lessons learned
```

## Senior Agent 입력 원칙

Senior Agent는 원문 전체를 읽지 않는다.

입력은 다음으로 제한한다.

```text
1. Current portfolio state
2. Reconciled cash/debt/tax reserve
3. Research structured packet
4. Macro regime summary
5. Risk Engine pre-check envelope
6. Relevant thesis memory
7. Last decision and outcome summary
```

## Delta-based Review

매주 전체를 다시 읽지 않는다.

매주 보는 것은 변화분이다.

```text
이번 주 바뀐 것:
- 가격 변화
- thesis-relevant news
- earnings/event update
- macro regime change
- correlation spike
- cash/ledger change
- risk rule breach
```

변화가 없으면:

```text
No material change since last review.
Use existing thesis status.
```

## Token Budget

초기 권장 예시:

```text
Research packet input: max 8,000 tokens
Senior memo input: max 6,000 tokens
Senior memo output: max 2,000 tokens
Telegram summary: max 500 tokens
```

초과 시:

```text
summarize by asset
keep only changes
retrieve only relevant memories
attach source links instead of full raw text
```

## Model Routing

| 작업 | 처리 방식 |
|---|---|
| 가격/비중 계산 | 코드 |
| CSV 파싱 | 코드 |
| 뉴스 분류 | 저렴한 모델/규칙 |
| 단순 요약 | 저렴한 모델 |
| Anti-thesis 추출 | 중간~고성능 모델 |
| Senior Memo | 고성능 모델 |
| Risk validation | 코드 |
| Telegram 요약 | 템플릿/저렴한 모델 |

## 10. 최종 문장

Portfolio OS의 목적은 시장을 예측하는 것이 아니다.

목적은 다음이다.

> 공격적인 장기 투자자가 자신의 확신, 감정, 편향, 장부 오류, 계산 실수, 과매매 충동, 세금 문제, 유동성 리스크, AI의 환각을 하나의 운영체계 안에서 통제하면서 더 나은 의사결정을 반복하도록 만드는 것.

따라서 이 프로젝트는 만들 가치가 있다.

하지만 반드시 이 순서로 만들어야 한다.

```text
AI부터 만들지 말 것.
Ledger부터 만들 것.
자동주문부터 만들지 말 것.
Manual execution + reconciliation부터 만들 것.
멋진 리포트부터 만들지 말 것.
정확한 장부와 Risk Engine부터 만들 것.
```

최종 원칙:

> **Ledger first.  
> Research must challenge conviction.  
> LLM judges, code calculates.  
> No sync, no official recommendation.  
> Human approves.  
> Overrides are allowed but must be declared.  
> The system exists to structure aggression, not to eliminate it.**
